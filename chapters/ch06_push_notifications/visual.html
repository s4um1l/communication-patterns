Payload Encryption (End-to-End)

1. Key Generation
Generate ephemeral ECDH key pair (P-256 curve)
2. Key Agreement
ECDH(ephemeral_private, subscription.p256dh) → shared_secret
3. Key Derivation
HKDF-SHA256(shared_secret, auth) → content_encryption_key + nonce
4. Encryption
AES-128-GCM(key, nonce, "Your food is arriving!") → ciphertext
5. HTTP POST
POST endpoint (ciphertext + ephemeral public key + VAPID JWT)

WebSocket (Ch05) vs Push Notification (Ch06)

WebSocket

Requires connection Yes (persistent TCP)
Delivery when offline No
Latency <1ms (after connect)
Direction Bidirectional
Server memory ~30KB/connection
Payload limit Unlimited
Best for Chat, real-time

Push Notification

Requires connection No (subscribe once)
Delivery when offline Yes (TTL-bounded)
Latency 1-10+ seconds
Direction Server → device only
Server memory ~200B/subscription
Payload limit 4 KB
Best for Alerts, offline reach
Key insight: Push notifications are the first pattern where the server can reach the client without ANY active connection. The trade-off is control: you delegate delivery to a third-party push service (Apple, Google, Mozilla), accept variable latency (1-10+ seconds), and encrypt every payload (because the push service is an untrusted relay). The server stores just ~200 bytes per subscription instead of maintaining a ~30KB WebSocket connection. For FoodDash, push handles "your driver is at the door" while WebSocket handles real-time chat. Both patterns coexist.
